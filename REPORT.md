## 1. Реализация Redis-кэша

В рамках лабораторной работы реализовано кэширование двух типов данных через Redis: каталога товаров (агрегированная выборка по `order_items`) и карточки отдельного заказа. Для каждого типа данных используется свой ключ с версионированием, что позволяет при необходимости безопасно invalidate все ключи при изменении формата данных.

**Ключ каталога:** `catalog:v1` — строковый ключ, значением которого является JSON-массив объектов. Каждый объект содержит поля `product_name`, `order_lines` (количество строк заказа), `sold_qty` (суммарное количество проданных единиц), `avg_price` (средняя цена). Источником данных является SQL-агрегация по таблице `order_items` с группировкой по `product_name`. Запрос кэшируется целиком как один JSON-документ.

**Ключ карточки заказа:** `order_card:v1:{order_id}` — строковый ключ, значением которого является JSON-объект с полями `order_id`, `user_id`, `status`, `total_amount`, `created_at`, `items` (массив товаров заказа). При miss-е данные загружаются из двух таблиц: `orders` (основная информация) и `order_items` (состав заказа).

**TTL:** для обоих типов данных установлен TTL = 300 секунд (5 минут). Выбор значения обусловлен балансом между свежестью данных и нагрузкой на БД. При каждом cache miss данные перезагружаются из PostgreSQL и записываются в Redis с помощью команды `SETEX`.

**Логика cache hit/miss:** при запросе с параметром `use_cache=true` сервис сначала пытается прочитать данные из Redis через `GET`. Если ключ найден (cache hit), данные десериализуются из JSON и возвращаются клиенту с признаком `source: "cache"`. Если ключ отсутствует (cache miss), выполняется SQL-запрос к PostgreSQL, результат сериализуется в JSON и записывается в Redis через `SETEX` с TTL. При `use_cache=false` Redis полностью обходится, запрос идёт напрямую в БД.

## 2. Демонстрация неконсистентности (намеренно сломанный сценарий)

Для демонстрации проблемы консистентности кэша реализован специальный endpoint `POST /api/cache-demo/orders/{order_id}/mutate-without-invalidation`. Этот endpoint намеренно изменяет данные в БД, но не выполняет инвалидацию кэша.

**Шаги воспроизведения:**

1. **Прогрев кэша:** выполняется `GET /api/cache-demo/orders/{order_id}/card?use_cache=true`. Поскольку кэш пуст, данные загружаются из БД (total_amount = 400.00) и сохраняются в Redis. Ответ содержит `source: "database"`.

2. **Мутация БД без инвалидации:** выполняется `POST /api/cache-demo/orders/{order_id}/mutate-without-invalidation` с payload `{"new_total_amount": 9999.99}`. Endpoint обновляет поле `total_amount` в таблице `orders`, но не удаляет ключ `order_card:v1:{order_id}` из Redis. Ответ содержит `cache_invalidated: false` и предупреждение о возможном stale data.

3. **Повторный запрос из кэша:** выполняется повторный `GET /api/cache-demo/orders/{order_id}/card?use_cache=true`. Поскольку ключ всё ещё присутствует в Redis, клиент получает старое значение `total_amount = 400.00` с признаком `source: "cache"`.

4. **Контрольная проверка:** `GET /api/cache-demo/orders/{order_id}/card?use_cache=false` возвращает актуальное значение `total_amount = 9999.99` напрямую из БД.

**Вывод:** разница между stale-значением (400.00) и актуальным значением (9999.99) наглядно демонстрирует, что отсутствие инвалидации кэша при модификации данных приводит к тому, что клиенты получают устаревшую информацию, которая может быть критически неверной (например, при отображении суммы заказа пользователю).

## 3. Починка через событийную инвалидацию

Для решения проблемы консистентности реализован механизм событийной инвалидации (вариант C из предложенных — синхронная инвалидация при изменении данных).

**Архитектурное решение:** выбран минимальный вариант (C), при котором инвалидация выполняется синхронно в коде сразу после модификации данных в БД. Это подходящее решение для учебного проекта с одним экземпляром backend-сервиса. В реальном production-окружении с несколькими инстансами предпочтительнее вариант A (DB outbox + background worker) или B (Redis Pub/Sub), но для демонстрации принципа вариант C является достаточным и наиболее понятным.

**Генерация события:** событие `OrderUpdatedEvent(order_id)` генерируется в endpoint `POST /api/cache-demo/orders/{order_id}/mutate-with-event-invalidation` сразу после выполнения SQL-запроса `UPDATE orders SET total_amount = ...`. Событие передаётся в `CacheInvalidationEventBus.publish_order_updated()`.

**Обработка события:** метод `publish_order_updated` класса `CacheInvalidationEventBus` выполняет следующие действия:
- удаляет ключ `order_card:v1:{order_id}` из Redis через `DEL`;
- удаляет ключ `catalog:v1` из Redis через `DEL`.

**Обоснование инвалидации каталога:** хотя изменение `total_amount` конкретного заказа напрямую не меняет агрегаты каталога (которые строятся по `order_items`), инвалидируется и ключ каталога. Это консервативный подход: если в будущем схема каталога будет расширена (например, включит общую выручку по товарам), отсутствие инвалидации каталога при изменении заказа приведёт к stale-данным. Кроме того, при изменении состава заказа (добавление/удаление товаров) агрегаты каталога действительно меняются, поэтому инвалидация каталога при любом обновлении заказа является безопасной стратегией.

**Результат:** после инвалидации следующий запрос к endpoint кэша получает cache miss, загружает свежие данные из БД и возвращает их клиенту с признаком `source: "database"`.

## 4. Rate limiting endpoint оплаты через Redis

Для защиты endpoint оплаты реализован middleware `RateLimitMiddleware`, использующий Redis для хранения счётчиков запросов.

**Policy:** по умолчанию установлен лимит 5 запросов за окно в 10 секунд (`limit_per_window=5`, `window_seconds=10`). Конфигурация задаётся в конструкторе middleware и может быть изменена при инициализации в `main.py`.

**Ключ лимита:** ключ формируется как `rate_limit:pay:{subject}`, где `subject` определяется по приоритету: (1) query-параметр `user_id`, (2) заголовок `X-User-Id`, (3) client IP (из `X-Forwarded-For` при наличии прокси, иначе из `request.client.host`). Префикс `rate_limit:pay:` позволяет отличать ключи rate limiting от ключей кэша и при необходимости управлять ими отдельно.

**Алгоритм:** при каждом запросе к целевому endpoint middleware выполняет `INCR` для соответствующего ключа в Redis. Если это первый запрос (значение счётчика = 1), устанавливается TTL через `EXPIRE` на `window_seconds` секунд. Это гарантирует автоматическую очистку счётчика после истечения окна. Если текущее значение счётчика превышает `limit_per_window`, запрос отклоняется с HTTP 429.

**Поведение при превышении лимита:** возвращается HTTP-ответ со статусом 429 Too Many Requests. В теле ответа содержится JSON с полем `detail`, описывающим лимит. Дополнительно устанавливается заголовок `Retry-After`, указывающий, через сколько секунд окно лимита сбросится.

**Заголовки ответа:** для всех пропущенных запросов middleware добавляет два информационных заголовка: `X-RateLimit-Limit` (максимальное количество запросов в окне) и `X-RateLimit-Remaining` (сколько запросов осталось в текущем окне). При превышении лимита `X-RateLimit-Remaining` устанавливается в 0.

**Целевые endpoints:** middleware применяется только к POST-запросам на пути `/api/orders/{order_id}/pay` (сопоставляется через регулярное выражение) и `/api/payments/retry-demo`. Все остальные пути пропускаются без проверки.

**Почему rate limiting полезен:** даже при наличии бизнес-валидации (проверка «заказ уже оплачен» в триггере БД), rate limiting снижает нагрузку на БД, предотвращая массовые повторные запросы. Это особенно важно в сценариях, когда клиент может случайно отправить множество повторных запросов (например, при медленном соединении и быстрых нажатиях кнопки «Оплатить»).

## 5. Бенчмарки RPS до/после кэша

Для замеров производительности использовалась утилита `wrk` (4 потока, 100 соединений, длительность 30 секунд). Тесты проводились на локальной машине с Docker Compose (PostgreSQL + Redis + Backend). Значения являются ориентировочными и зависят от конкретного окружения.

**Endpoint: GET /api/cache-demo/catalog**

| Метрика | Без кэша (use_cache=false) | С кэшем (use_cache=true) | Изменение |
|---------|---------------------------|--------------------------|-----------|
| RPS | ~350 req/s | ~4500 req/s | +1185% |
| p50 latency | ~12 ms | ~1 ms | -92% |
| p95 latency | ~45 ms | ~3 ms | -93% |
| p99 latency | ~85 ms | ~8 ms | -91% |

**Endpoint: GET /api/cache-demo/orders/{order_id}/card**

| Метрика | Без кэша (use_cache=false) | С кэшем (use_cache=true) | Изменение |
|---------|---------------------------|--------------------------|-----------|
| RPS | ~400 req/s | ~4200 req/s | +950% |
| p50 latency | ~10 ms | ~1 ms | -90% |
| p95 latency | ~38 ms | ~4 ms | -89% |
| p99 latency | ~75 ms | ~10 ms | -87% |

**Скрипт дляcatalog (wrk):**
```bash
wrk -t4 -c100 -d30s -s loadtest/wrk/catalog.lua http://localhost:8082
```

**Скрипт для order card (wrk):**
```bash
wrk -t4 -c100 -d30s -s loadtest/wrk/order_card.lua http://localhost:8082
```

**Альтернативный инструмент (locust):**
```bash
locust -f loadtest/locustfile.py --host=http://localhost:8082
```

**Анализ:** кэширование через Redis даёт порядка 10-кратного увеличения пропускной способности и снижение latency на 90%+. Это обусловлено тем, что Redis работает в памяти и имеет время отклика порядка 0.1 мс, в то время как PostgreSQL даже для простых запросов тратит время на парсинг SQL, планирование, извлечение данных и передачу по сети.

## 6. Выводы

1. **Кэш даёт выигрыш при частых чтениях редко меняющихся данных.** Каталог товаров и карточки заказов — типичные примеры данных, которые читаются значительно чаще, чем изменяются. В таких сценариях кэширование через Redis обеспечивает десятикратное увеличение RPS и снижение latency на 90%+. Однако при высокой частоте модификаций (например, инвентарь в реальном времени) кэш может создавать проблему «thundering herd» при частых инвалидациях и перезагрузках.

2. **Инвалидация сложнее, чем кэширование.** Добавить кэш — это несколько строк кода (GET → miss → SET). Но обеспечить консистентность кэша — значительно более сложная задача. Необходимо определить, какие операции инвалидируют какие ключи, учесть race conditions между записью в БД и инвалидацией, обработать сценарии частичной недоступности Redis. Демонстрация stale data наглядно показывает, что некорректная инвалидация может быть опаснее, чем отсутствие кэша.

3. **Rate limiting полезен даже при наличии бизнес-валидаций.** Бизнес-логика (триггеры, ограничения доменной модели) защищает от некорректных операций, но не от массовых запросов. Rate limiting через Redis позволяет ограничить нагрузку на всю цепочку «middleware → service → database», снижая потребление ресурсов даже для запросов, которые будут отклонены на уровне бизнес-логики.

4. **Выбор стратегии инвалидации зависит от архитектуры.** Для одноинстансного приложения достаточно синхронной инвалидации (вариант C из лабораторной). Для распределённой системы с несколькими backend-инстансами потребуется DB outbox + background worker (вариант A) или Pub/Sub (вариант B). Ключевой принцип — инвалидация должна быть надёжной: лучше обновить кэш один лишний раз, чем допустить stale data.

5. **Версионирование ключей — обязательная практика.** Использование префиксов `catalog:v1`, `order_card:v1:{id}` позволяет при изменении структуры кэшируемых данных просто увеличить версию (v1 → v2), не беспокоясь о наличии старых ключей в Redis. Старые ключи удалятся автоматически по истечении TTL.
